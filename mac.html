四、Git --分布式版本控制系统 （2005年首发）

1.集中式vs分布式

集中式：

①必须联网

②版本集中存放在中央服务器

③工作时需从中央服务器下载，完成后重新提交给中央服务器

分布式：

①不用联网

②本地有完整的版本库

③安全性较高

2.安装Git

①brew install git

②从Appstore安装集成了Git的Xcode

3.创建版本库 (删除版本库 find . -name “.git” | xargs rm -Rf)

①创建一个空目录 

mkdir xx

cd xx

pwd

②将xx目录变成Git可以管理的仓库

git init 

③查看目录中是否有.git目录，默认隐藏，可用ls -ah

4.将文件添加到版本库

①添加文件 git add xx （添加多个文件 git add 1.txt 2.txt）

（添加目录及内容 git add 目录/）

（添加目录中所有某种类型的文件 git add *.文件类型）

②把文件提交到仓库 git commit -m “说明信息”

5.修改文件后的处理方法

①修改文件后，运行git status 查看仓库状态，提示需要提交，执行git commit -m “message”

②不记得上次修改，可执行git diff xx 查看上次修改的记录

  git diff HEAD -- xx 可查看某xx文件在工作区与版本库里最新版本的区别

6.mac中每个目录都会有.DS_store，因此提交时需要忽略

①创建.gitignore文件，用于忽略文件，其规范如下

1.所有空行或者以注释符号 ＃ 开头的行都会被 Git 忽略。
 2.可以使用标准的 glob 模式匹配：

①* 任意个任意字符）

②[] 匹配任何一个在方括号中的字符

③? 匹配一个任意字符

④[0-9] 匹配字符范围内所有字符
3.匹配模式最后跟反斜杠（/）说明要忽略的是目录。
4.要忽略指定模式以外的文件或目录，可以在模式前加上惊叹号（!）取反

①failed to push some refs to...  因为远程库与本地库不一致造成的

解决方法：把远程库同步到本地库

git pull --rebase origin master

8.版本回退

①git log --pretty=oneline 查看提交日志

日志中有commit id

②回退到上一个版本 git reset --hard HEAD^

(当前版本 HEAD  上一个版本 HEAD^  上n个版本 HEAD~n)

③回退到指定版本 git reset --hard xxxx(commit id 的前几位)

 

④误操作，可用 git reflog 查看之前的所有操作(里面有commit id)

 

9.撤销修改

①丢弃工作区的修改 git checkout -- xx

当xx修改后没有放到暂存区中，撤销修改，工作区与版本库保持一致

当xx已经放到暂存区中,又作了修改,在撤销修改,工作区与暂存区保持一致

②当修改只是添加到了暂存区，还没有提交

可用git reset HEAD xx把暂存区的修改回退到工作区 

10.删除操作

①删除工作区文件rm xx, 可用 git checkout -- xx恢复(与版本库中最新版本一致)

②删除版本库文件 git rm xx ,提交修改 git commit

11.添加远程仓库

①创建SSH Key

ssh-keygen -t rsa -C “邮件地址”

用户主目录里找到.ssh目录，里面有id_rsa(私钥)和id_rsa.pub(公钥)

查看cat ~/.ssh/id_rsa.pub中的公钥

②登录gitlab账户，搜索SSH Key ,粘贴id_rsa.pub 后提交即可

③在gitlab上新建仓库（远程库的名字默认是origin）

④在本地仓库运行 git remote add origin git@gitlab.com:用户名/远程库文件

⑤把本地库的所有内容推送到远程库中 git push -u origin master (-u 可将本地分支master 与远程的master分支关联起来)

⑥本地提交到远程库 git push origin master

12.克隆一个本地库

git clone git@gitlab.com:用户名/远程库文件

13.分支管理

①创建分支

git checkout -b dev 创建dev分支，并切换(-b)到dev分支

等效于 创建分支git branch dev 和切换分支 git checkout dev

②查看分支状态

   git branch 

③dev分支的工作完成，切回master分支

   git checkout master

④将de分支与master 分支合并

   git merge dev（快速合并）

⑤合并后可删除dev分支

git branch -d dev

⑥当master分支和feature分支各自都分别有新的提交，无法快速合并

git merge feature 提示Merge conflict in xx文件

git status 查看冲突的文件

open xx 打开冲突的文件，修改冲突后，

再提交（git add xx  git commit -m “message”）

之后就可以删除feature分支（git branch -d feature）  

        

⑦禁用快速合并，可用 git merge --no-ff -m “message” feature

14.Bug分支

①在修复bug的时候，可以当前dev分支的工作现场储藏起来(git stash)

②git status 查看工作区，如果没有提示需要管理的文件，就可以建分支修复bug

③确定bug在那个分支中

假定bug在master, git checkout master

创建issue-101分支并切换git checkout -b issue-101 

open bug文件，修改完，提交

git add xx  git commit -m “fix bug 101”

修复完，切回master，并完成合并，最后删除分支

git checkout master 

git merge --on-ff -m “message” issue-101

git branch -d issue-101

④接着回到dev分支，继续工作

git checkout dev 切回dev分支

git status 后发现工作区是干净的

git stash list 查看之前储藏的工作现场

这时需要恢复工作现场，

一是git stash apply恢复，stash内容不会删除，删除的话，可用git stash drop

另一种用git stash pop恢复，恢复的同时把stash内容删除

 

当多次stash，恢复的时候，先用git stash list查看，确定一个stash

git stash apply stash@{0}

⑤添加新功能时，需创建分支用于开发与测试

git checkout -b feature-music

开发完毕

git add music.py

git status 看是否都添加了,确认无误后，提交

git commit -m “add feature music”

切回dev，准备合并

git checkout dev

git merge --on-ff -m “message” feature-music

git branch -d feature-music

⑥新功能需要撤销，feature-music还没有合并，

这时执行 git branch -d feature-music 

会提示feature-music还没有合并，删除将丢失修改

此时要想强行删除，可用 git branch -D feature-music

15.多人协作（远程仓库默认名称是origin）

①从远程仓库clone时，就将本地的master和远程的master分支关联起来

②查看远程仓库的信息 git remote -v (-v 表示详细信息)

③推送分支 git push origin master

  推送dev分支 git push origin dev

④master分支是主分支，需时刻与远程同步

  dev分支是开发分支，团队所有成员都在上面工作，也需要与远程同步

  bug分支只用于本地修复，无需与远程同步

  feature分支有合作的话，与远程同步

⑤抓取分支

从远程仓库clone时，默认只clone master

要想看到其他分支的步骤如下：

创建远程origin的dev分支到本地 

git checkout -b dev origin/dev

修改(git add xx  git commit -m “add xx”)完，

把dev分支push到远程仓库

git push origin dev

⑥当你的组员向origin/dev分支推送他的提交，碰巧你也对相同的文件做了修改， 这时推送会提示文件冲突，解决办法如下：

首先从origin/dev抓取最新提交

（

需提前将本地dev和远程origin/dev分支关联起来，

没有关联的话，git pull报错

git branch --set-upstream-to=origin/dev dev

）

git pull 后会提示那个文件冲突，将文件手动合并后

本地提交 git add xx   git commit -m “fix xx conflict”

推送到远程仓库 git push origin dev

         ⑦多人在同一个分支上协作时，很容易出现冲突，每次合并在push，会导致提交 历史很乱，这时可用 git rebase 将提交历史变成一条直线（会修改本地的分 叉提交）

16.标签管理

①创建标签（标签总是和某个commit挂钩）

首先切换到需要打标签的分支上，

git branch 查看分支，切换分支 git checkout master

然后，执行 git tag v1.0 打上标签（默认标签打在最新提交的commit上）

可用 git tag 查看所有的标签

②忘记打标签，可用git log --pretty=oneline --abbrev-commit找到历史提交的 commit id，执行 git tag v0.9 f25342

③还可以创建带有说明的标签（-a 指定标签名 -m 指定说明文字）

git tag -a v0.1 -m “version 0.1 released” 1094adb

git show v0.1 可以查看说明文字

④操作标签（创建的标签都只存储在本地，不会自动推送到远程）

本地删除标签 git tag -d v0.1

推送某个标签到远程 git push origin v1.0

也可以一次性推送所有的本地标签 git push origin --tags

⑤标签已经推送到远程，如何删除

首先删除本地标签 git tag -d v0.9

然后，从远程删除 git push origin :ref/tags/v0.9
